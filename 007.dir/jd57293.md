What this testcase tests:
Respecting loaded segments from the ELF loader

How this testcase verifies intended behavior:
This testcase forcefully puts its program's segments at an address conflicting with the hardcoded address in the given code and expects the program to still run correctly

Detailed explanation of testcase:
In the given code, `kernel_main.cc` hard codes the user stack's address and disregards the sections that the ELF loader mapped to. This test case includes a very simple cpp program and uses a custom linker script `custom.ld` to override the default linker and forcefully start the program's sections at `0x0007fffffe8000` which overlaps with the stack range (`0x7ffffff0000` - 64 KiB). In the starter code, map_range will overwrite the program's only LOAD segment with the stack region and cause the test to fail. 

How to solve:
To properly solve this issue, the kernel needs to track where the ELF loader is putting program segments and respect those boundaries which can be done in lots of different ways
(Technically you could just change the user rsp to another hardcoded value that doesn't clobber the program segment and also pass this specific test...)


Explanation of modifications to non-cpp files:
- `custom.ld`: simple linker script to start loading program segments at a specified address
- `Makefile`: modified to use linker script, keep GCC happy, and copy binary to sbin/
